API Planning

API Planning Framework

How to decide what endpoints to build — UI-first approach.

1

Start from the Screen

List every screen in the Mobile App and Web Admin. Each screen = a data requirement.

→
2

Map Data Needs

For each screen, write what data it reads (GET) and what it changes (POST/PUT/DELETE).

→
3

Derive Endpoints

Only create an endpoint when a screen needs it. Delete or defer any placeholder with no screen owner.

⚠️
Golden Rule:

If no screen in Mobile or Web Admin calls an endpoint, mark it DEFERRED and do not spend time on it.

Immediate Action: API Audit

Export list of ALL current API endpoints from the .NET project
Against each endpoint, write which screen uses it (Mobile or Admin)
Any endpoint with no screen owner → mark PLACEHOLDER / DELETE
Any missing endpoint a screen needs → mark TODO / SPRINT BACKLOG

On-Road Unit App

On-Road Service Unit App

Flutter app for wash technicians — scoped to the minimum needed for MVP.

ℹ️

Per MVP spec, this is a lightweight Flutter app for on-road units. No complex features — just receive assigned jobs, update status at key moments, and submit a completion checklist with before/after photos. The vendor admin assigns jobs; this app is the technician's tool.

App Screens
🔐

Login

Phone + OTP authentication — reuses the same auth API as the customer app.

📋

Job Queue

List of bookings assigned by the vendor admin, ordered by scheduled time.

🚗

Job Detail

Customer info, vehicle details, service address, package type, and special requests.

📍

Status Updates

Simple button flow: En Route → Arrived → In Progress → Completed

✅

Completion Checklist

Tick required items + upload before/after photos to Azure Blob before submitting.

🔔

Push Notifications

Alert when a new job is assigned by the vendor admin. No in-app messaging needed.

Execution

Git Workflow & CI/CD

Lean branching strategy for a 4-person team using GitHub + GitHub Actions + Azure.

Branching Strategy
main
Production. Protected. Merge via PR only. No direct pushes allowed.
develop
Integration branch. All features merge here first before going to main.
feature/*
One branch per task. e.g. feature/booking-api
hotfix/*
Emergency production fixes only. Branch from main, merge to main + develop.
GitHub Actions Pipeline
1

Push / PR

Triggers on push to feature/* or PR opened to develop

2

Build + Test

.NET: dotnet build + test
Angular: ng build
Flutter: flutter test

3

PR Review

Senior or Mid must approve PR before merge to develop

4

Deploy Staging

Auto-deploy to Azure (App Service + Static Web App) on merge to develop

5

Deploy Prod

Manual trigger. PR from develop → main + required approval

No direct push to main or develop
All PRs need 1 review minimum
Failing tests block merge
Secrets stored in GitHub Secrets — never in code
